## CRITICAL (1 — fixed)

### caja: `crear()` usaba I/E con importe negativo, contaminaba data legacy

**Síntoma:** Las filas escritas por el app entre Apr 17–Apr 30 tenían
`tipo='I'` o `tipo='E'` con `importe<0`. El DB legacy (722 rows) usa
`tipo='E'/'S'` con `importe` siempre positivo. `resumen()` lee la
convención legacy. Resultado: los movimientos del nuevo app no aparecían
correctamente en los totales — un egreso entrado por la UI se contaba
como una "entrada negativa". El saldo aritmético quedaba bien por doble
negativo, pero los desgloses ingresos/egresos eran basura.

**Fix (commit, 2026-04-30):**
- `modules/caja/queries.py::crear()` ahora valida `tipo in ('E','S','CB')`,
  toma `abs(importe)` (siempre positivo), y firma con `tipo` para chequear
  saldo no-negativo. Docstring del módulo y de la función documenta la
  convención.
- `modules/caja/queries.py::saldo_actual()` reescrita para usar
  `opening + Σ E − Σ S` en una sola query (antes hacía `SUM(importe)`
  pelado).
- `modules/caja/views.py` cambia el dominio de `tipo` de `('I','E')` a
  `('E','S')`.
- `modules/caja/templates/caja/lista.html`:
  - botones "+ Entrada" → `?tipo=E`, "+ Salida" → `?tipo=S`.
  - render del tipo cambia a chip Entrada/Salida en lugar de la letra.
  - la columna importe pinta verde/rojo y muestra signo `−` según `tipo`,
    no según el sign de `importe`.
- `modules/caja/templates/caja/nuevo.html` select reemplaza
  Ingreso/Egreso por Entrada/Salida con value E/S.

**Migración pendiente (manual, una sola vez):** las filas escritas por el
app buggeado (Apr 17–Apr 30) tienen tipo='I' o tipo='E' con importe<0.
Limpiar con:

```sql
-- Filas tipo='I' (ingreso buggy) → convertir a 'E' (entrada legacy).
UPDATE scintela.caja
   SET tipo = 'E',
       importe = ABS(importe)
 WHERE tipo = 'I'
   AND importe IS NOT NULL;

-- Filas tipo='E' con importe negativo (egreso buggy) → convertir a 'S'.
UPDATE scintela.caja
   SET tipo = 'S',
       importe = ABS(importe)
 WHERE tipo = 'E'
   AND importe < 0;
```

Después correr `recompute_saldos_desde(min(fecha_buggy))` para que el
running balance quede consistente.

`python scripts/migrate.py --force 0003`.

### periodo_guard.py — fallo silente si el módulo periodos no carga

**Síntoma:** El `try/except` en el import era amplio y no logueaba.
Si `modules.periodos` rompía por syntax error, import circular o
DB ausente, `_periodos_q = None` y `asegurar_fecha_abierta()` se volvía
no-op. Todas las escrituras a períodos cerrados pasaban silenciosamente.

**Fix:**
- `logger.exception(...)` cuando el import falla (queda en
  `/tmp/programa-core.log`).
- Variable de entorno `PERIODO_GUARD_STRICT=1` que hace que la función
  levante `RuntimeError` en lugar de no-op cuando el módulo no cargó.
  Recomendado en producción una vez confirmado que periodos siempre carga.

### caja: `saldo_actual()` antes hacía `SUM(importe)` plano

**Síntoma:** Mismo patrón del bug del audit-warning. La función se usa
en `crear()` para chequear overdraft, así que el chequeo de saldo
negativo era poco confiable durante el período buggy.

**Fix:** Reescrita para usar `opening + Σ entradas − Σ salidas` con
`CASE WHEN tipo`.

## CRITICAL #2 — historia.stock (kg) leído como historia.ustock (US$)

Encontrado por el usuario el 30/4 al ver el balance: la columna izquierda
de STOCK total ($9.604.034) y la derecha de STOCK MP+PROD ($8.097.939)
no cuadran. Diff $1.506.095. Investigando, son DOS bugs distintos:

### Bug A — `stock_kg_live()` mezclaba kg y US$

`modules/informes/queries.py::stock_kg_live()` línea 572 tenía:

```python
snap_kg = float(hist.get("ustock") or 0)  # leía US$, lo llamaba kg
```

Pero `historia.ustock` está en US$, mientras que `historia.stock` está
en kg (las columnas con prefijo `u` son US$, sin prefijo son kg). El
comentario de líneas 1097-1100 lo advertía explícitamente:

> "OJO: VSTO se guarda en `historia.ustock` (en US$). `historia.stock` es
> otra columna (kg, no $) y leerla acá nos costó ~5.8M de patrimonio
> mid-mes."

La función después sumaba kg comprados/vendidos a este valor en US$:

```python
"live_kg": snap_kg + kg_com - kg_ven,  # mezcla US$ + kg!
```

**Fix:** Leer `hist.get("stock")` (kg) y actualizar docstring.

### Bug B — `informe_balance()` línea 1118: mismo error

```python
kg = {
    "stock_kg": float(hist.get("ustock") or 0),  # label dice kg, valor en US$
    "stock_kg_diff": live["live_kg"] - float(hist.get("ustock") or 0),
}
```

**Fix:** Misma corrección — leer `stock` en lugar de `ustock`.

### Bug C — Discrepancia de DISEÑO entre STOCK izq y der

Después de fixear A y B, queda la diff de $1.5M entre las dos vistas
del stock. Es un problema de DISEÑO, no de código:

- **IZQUIERDA** (`r.stock.total.us` = $9.604.034): live, computado como
  `kg(historia) × $/kg(iniciales)`. El `$/kg` viene de `iniciales.um`,
  `iniciales.uk`, `iniciales.uf` — **rates de apertura del período**.
- **DERECHA** (`b.vsto` = $8.097.939): snapshot, leído directamente de
  `historia.ustock` — el valor **congelado al último cierre**. Este es
  el que feeds PATRIMONIO.

Cambiar `vsto` a la versión live alteraría el patrimonio reportado, lo
cual es financialmente sensible. Solución prudente: anotar la diff
explícitamente en la UI para que el gerente sepa qué representa cada
uno y cuál es canónico.

**Fix:**
- En el balance template, agregada fila amber abajo del Total de la
  izquierda explicando la diff y aclarando que **el canónico para
  patrimonio es STOCK MP+PROD (vsto)**.
- En el ACTIVO derecho, agregado tooltip explicativo en STOCK MP+PROD.
- Tests de `stock_kg_live` actualizados para usar el nombre correcto
  de columna (`stock`, no `ustock`) en los fixtures.

### Lección — separar prefijos `k...` (kg) y `u...` (US$)

Las columnas de `historia` siguen una convención estricta:
- Sin prefijo o con `k`: kilos. Ej: `stock`, `kcom`, `kvent`.
- Con prefijo `u`: US$. Ej: `ustock`, `ucom`, `uvent`.

Para código nuevo, **NUNCA escribir `hist.get("u" + algo)` cuando se
necesita kg, ni viceversa**. Pequeño decorator/helper podría detectar
esto (anotación de tipos), pero alcanza con leer dos veces antes de
poner el nombre.

## Lección de meta-proceso

Despachar 4 audits paralelos (SQL / transactions / permissions / templates)
fue rápido pero los agentes erraron en 2 puntos por leer parcial. Cuando
un audit dice "X no tiene Y", verificar leyendo el archivo. La auditoría
sirve para encontrar candidatos, no para decisiones finales. Cada hallazgo
tiene que ser confirmado leyendo el código antes de gastar trabajo en él.

El bug crítico (caja tipo) lo encontró sólo después de pegar el caso —
"resumen() usa E/S pero crear() usa I/E" — al leer dos funciones del mismo
archivo en secuencia. El audit agente había marcado todo como GOOD en caja
porque no comparó las dos convenciones entre sí.
