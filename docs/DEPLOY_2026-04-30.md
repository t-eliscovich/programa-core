# Deploy 2026-04-30 — checklist de migraciones pendientes

Tres migraciones pendientes en producción (RDS). Aplicar **antes** del
próximo ship de código.

## Orden de pasos

```bash
# 1. Verificar qué falta (read-only, no toca nada)
python scripts/migrate.py --status

# Esperado: 0006, 0007, 0017 deberían aparecer como "pending" si no se
# aplicaron. Si ves más, son cambios que no recordamos — investigar
# antes de seguir.

# 2. Backup defensivo de scintela.caja antes de tocar tipo/importe
#    (la migración 0017 es destructiva sobre data del Apr 17–30)
psql $DATABASE_URL -c "
    CREATE TABLE IF NOT EXISTS scintela.caja_backup_pre_0017 AS
    SELECT * FROM scintela.caja;
"

# 3. Aplicar las migraciones pendientes
python scripts/migrate.py

# 4. Verificar que ninguna fila quedó con tipo='I' o importe<0
psql $DATABASE_URL -c "
    SELECT COUNT(*) AS n_buggy_tipo FROM scintela.caja WHERE tipo='I';
    SELECT COUNT(*) AS n_buggy_neg  FROM scintela.caja WHERE importe<0;
"
# Esperado: ambas queries devuelven 0. Si alguna devuelve >0 la migración
# se abortó (tiene un DO $$ que tira EXCEPTION) — investigar el log y
# restaurar de scintela.caja_backup_pre_0017 si hace falta.
```

## Qué hace cada migración

### 0006 — bitacora_request_id
Agrega columna `request_id VARCHAR(36)` a `scintela.bitacora_acciones`
para correlacionar logs con renglones de auditoría. Sin esto, el código
nuevo del `auth.py::registrar_bitacora` falla el INSERT y se traga el
error (queda audit incompleto).

### 0007 — ejecuciones_tareas
Crea tabla tracker `scintela.ejecuciones_tareas` con UNIQUE(tarea, periodo)
para tareas programadas at-most-once-per-period. La usa
`scripts/procesa_provisiones_mensual.py` (cron mensual). Sin esto, el
cron sale con exit 2.

### 0017 — fix_caja_tipo_buggy
Limpia las filas de `scintela.caja` escritas entre Apr 17–30 con la
convención de tipo INCORRECTA (tipo='I' o tipo='E' con importe<0). Las
convierte a la convención legacy E/S con importe siempre positivo, y
recalcula el running `saldo` desde el opening hacia adelante.

Hace tres pasos en una sola transacción:
1. `UPDATE` filas tipo='I' → tipo='E' con `ABS(importe)`.
2. `UPDATE` filas tipo='E' AND importe<0 → tipo='S' con `ABS(importe)`.
3. Recalcula running `saldo` con window function partiendo del opening.
4. Verifica que ninguna fila quedó violando el invariante (tipo='I' o
   importe<0) — si alguna queda, `RAISE EXCEPTION` y rollback.

Idempotente: el segundo run encuentra 0 filas que matchean el WHERE y
es no-op.

## Después del deploy

1. **Rotar Scheduled Task de provisiones** — registrar el cron mensual
   con el PowerShell del skill `intela-aws-deploy` (sección "Programa
   Core — tarea mensual de provisiones"). Esto sólo después de que 0007
   esté aplicado en producción.
2. **Smoke test rápido**: abrir `/caja` en el app de producción y
   verificar que el "Saldo en caja" coincide con `opening + Σ E − Σ S`
   sin warning de auditoría. Y que las nuevas entradas que crees usen
   tipo='E' o 'S' (no 'I' ni importe negativo).
3. **Verificar bitácora**: hacer cualquier write (crear un cheque de
   prueba), y ver que la bitácora muestre `request_id` con UUID
   completo. Header `X-Request-Id` debería aparecer en cualquier
   response.

## Rollback (si algo se rompe)

```bash
# Restaurar caja desde backup
psql $DATABASE_URL -c "
    BEGIN;
    DELETE FROM scintela.caja;
    INSERT INTO scintela.caja
        SELECT * FROM scintela.caja_backup_pre_0017;
    COMMIT;
"

# Quitar las migraciones de la tabla tracker para que el runner las
# vuelva a intentar la próxima
psql $DATABASE_URL -c "
    DELETE FROM seguridad.migraciones_aplicadas
     WHERE version IN ('0006', '0007', '0017');
"
```

Si el rollback es necesario, abrir un addendum `SKILL_ADDENDUM_BATCH_24.md`
documentando qué falló para que el siguiente intento sepa qué evitar.
