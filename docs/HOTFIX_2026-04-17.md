# Hotfix 2026-04-17 — primera corrida post-batch-12

Cuando te levantes, leé esto antes de seguir.

## Qué pasó

Arrancaste el app con `launcher.sh` y te topaste con 3 errores, en orden:

1. **`pip install` falló** con `psycopg-binary==3.2.3 no existe`.
2. **`Bad Request: CSRF token missing`** al hacer login.
3. **`Internal Server Error`** después del login.

Los tres ya están arreglados. Abajo está el detalle de cada uno.

## Los 3 fixes

### Fix 1 — `requirements.txt`

`psycopg[binary]==3.2.3` no existía en PyPI. Lo cambié a `psycopg[binary]>=3.2.10`
(primera versión real). Esto es para `pytest-postgresql`, que sólo se usa en los
integration tests `@pytest.mark.db`; la app en runtime sigue usando `psycopg2-binary`.

### Fix 2 — CSRF tokens faltantes

`login.html` y el form de logout en `base.html` nunca tuvieron el hidden input
`<input name="csrf_token" value="{{ csrf_token() }}">`. Cuando batch 9 activó
CSRFProtect globalmente, se olvidó actualizar estos dos templates. Agregado.

### Fix 3 — el 500 después del login

**Causa raíz**: la query de login en `auth.py` hace:

```sql
SELECT id_usuario, username, password_hash, activo,
       totp_secret, totp_confirmado_en, password_debe_cambiar
FROM seguridad.usuario
```

Las últimas 3 columnas (`totp_secret`, `totp_confirmado_en`,
`password_debe_cambiar`) las agregó la **migración 0008**. Tu base de datos local
no tiene 0008 aplicada — probablemente la última vez que corriste migrate.py fue
antes de batch 8. El SELECT tira `UndefinedColumn`, Flask lo convierte en 500.

**Fix aplicado**: el SELECT ahora es defensivo. Intenta primero con las cols nuevas;
si la DB levanta `UndefinedColumn`, hace fallback a un SELECT básico con defaults
razonables (`totp_confirmado_en=None`, `password_debe_cambiar=False`). La app
funciona sin 2FA hasta que apliques la migración.

Lo mismo con `cambiar_password` (otra query que lee `password_debe_cambiar`).

**Regression test agregado**: `tests/test_auth.py::test_login_funciona_sin_migracion_0008`
que monkey-patchea `db.fetch_one` para simular la falla. 294 tests verde.

## Qué corrés vos mañana

**Una vez**, en la terminal (dentro del proyecto con venv activo):

```bash
python scripts/migrate.py
```

Eso aplica las migraciones pendientes (0006, 0007, 0008, 0009). Output esperado:

```
Aplicando 0006_bitacora_request_id.sql … OK
Aplicando 0007_ejecuciones_tareas.sql … OK
Aplicando 0008_2fa_y_password_policy.sql … OK
Aplicando 0009_factura_electronica.sql … OK
```

**Después** (aún sin aplicar la migración, el app funciona):

```bash
./launcher.sh
```

Y abrís `http://localhost:5050/login`. El login con usuario/password del `.env`
debería dejarte entrar.

## Si `migrate.py` falla

Suele ser que `seguridad.migraciones_aplicadas` está inconsistente. Chequeo:

```bash
psql -U intela -d intela -c "SELECT version, nombre, aplicada_en FROM seguridad.migraciones_aplicadas ORDER BY version"
```

Las 5 primeras (0001-0005) deberían estar. Si falta alguna, corré:

```bash
python scripts/migrate.py --status
```

y seguís las instrucciones.

## Si no podés loguear

- Verificá que el usuario existe: `psql -U intela -d intela -c "SELECT username FROM seguridad.usuario"`.
- Si la tabla está vacía, corré `python scripts/seed_roles.py` para crear el admin del `.env`
  (`tamara / intela2026` por default).
